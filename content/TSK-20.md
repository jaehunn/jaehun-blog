# [시스템 디자인] 사용자 규모와 확장성 1

> 이 글은 “가상 면접 사례로 배우는 대규모 시스템 설계 기초” 책을 재구성한 글입니다.

몇 백만의 사용자를 지원하는 서비스들은 어떻게 설계되어있을까?

간단한 시스템 설계부터 차근차근 한 번 알아보자.

## 1. 사용자의 요청이 처리되는 과정

시스템을 설계하려면, 먼저 시스템이 어떻게 동작하는지 이해할 필요가 있다. 아래는 사용자 요청을 처리하는 흐름을 나타낸 그림이다.

![1.png](/images/TSK-20/1.png)

- 사용자 (Client) 는 [https://www.example.com](https://www.example.com) 에 접속하기 위해 IP 주소를 DNS (Domain Name Service) 에 질의하여 알아낸다. DNS 는 제3 사업자 (Third Party) 에서 제공하는 유료 서비스를 이용하므로 우리가 설계하는 시스템의 일부는 아니다.
- DNS 에서 알아낸 IP 주소는 [https://www.example.com](https://www.example.com) 의 Web Server 주소다. Web Server 로 HTML 또는 JSON 형태의 응답을 받기 위해 HTTP (HyperText Transfer Protocol) 요청이 전달된다.

> Client → DNS → Client → Web Server

수 많은 사용자를 대응하기 위해서는 Web Server 를 여러개 두어야하는데,

이 때, 관심사를 분리할 수 있도록 레이어링을 하면 독립적으로 웹 서버를 확장시킬 수 있다.

- (1) 트래픽을 처리하는 레이어
- (2) 데이터를 처리하는 레이어

## 2. 확장의 개념, 스케일 업 vs. 스케일 아웃

Scale Up 이라고 하는 개념은, 더 좋은 사양의 서버를 사용하는 것을 말한다. (더 좋은 CPU, 더 많은 메모리.)

반면 Scale Out 은 수평 확장이라고 하며, 서버의 수를 늘리는 것을 말한다.

서버로 유입되는 트래픽 (사용자의 접근) 이 적을 때는 서버의 성능을 향상시키는 게 좋다.

이 방법은 가장 단순하지만, 몇가지 단점이 있다.

- 메모리나 CPU 를 무한정으로 늘리는 것이 불가능하다.
- 또 업그레이드 된 서버가 죽었을 때 대응방안이 불명확하다. (그대로 서비스는 중단된다.)

대규모의 서비스를 지원하려면, 수평적 확장이 보다 적절하다.

또 스케일 업이 가지는 몇가지 단점을 보완한다. 이 보완법을 한 번 알아보자.

## 3. 로드밸런서 (LB, Load Balancer)

Load Balancer 는 여러 서버들에게 트래픽을 고르게 분산하는 역할을 한다.

![2.png](/images/TSK-20/2.png)

- Client 가 DNS 로 받아낸 IP 주소는 LB 의 주소가 된다. 이때 LB IP 는 Public IP 로 구성된다.
- LB 는 Private IP 로 웹 서버들과 통신한다. 또 웹 서버들은 Private 한 IP 로 통신한다.

LB 가 제시하는 부하 분산은

- 서버가 죽었을 때, 다른 서버로 트래픽을 전송시켜 완전히 서비스가 중단되는 상황을 방지한다. → 자동 복구(Failover)
- 트래픽이 가파르게 증가하면, 자동적으로 트래픽을 고르게 분산하기 위해 서버를 유동적으로 추가할 수 있다. → 가용성 (Availability)

트래픽을 처리하는 레이어로 LB 를 알아보았다.

그러면, 데이터를 처리하는 레이어는 어떻게 최적화시킬 수 있을까?

## 4. 데이터베이스 (Database)

데이터베이스는 두가지가 있다.

- 데이터가 관계를 가지는 Relational Database
- 데이터가 관계를 가지지 않는 NoSQL Database

관계를 가지는 데이터베이스는 MySQL, Oracle Database, PostgreSQL 등이 있다. 데이터의 집합을 테이블로 표현하고, 테이블의 관계에 따라 Join 하여 합쳐 사용하도록 한다.

NoSQL Database 는 Cassandra, DynamoDB 등이 있다. 단순하게 Key-Value 저장소로 사용되고, Join 과 같은 테이블 연산은 지원하지 않는다.

일반적으로 관계형 데이터베이스를 사용하지만, 아래와 같은 경우에는 NoSQL 을 고려해보면 좋다.

- 사용자에게 빠른 응답을 제공해야하는 경우 (Low Latency)
- 데이터가 비정형일때 (Unstructured)
- 데이터를 직렬화(Serialize), 역직렬화(deserialize) 만 하면 될 때
- 아주 많은 양의 데이터를 저장해야 할 때

많은 데이터베이스 시스템은 다중화를 지원한다.

데이터를 Master-Slave 관계로 설정하고, 원본은 Master, 사본은 Slave 에 설정한다.

쓰기 연산(Write Operation) 은 Master 에서만 지원하며, Slave 는 Master 로 부터 사본을 전달받고, 읽기 연산(Read Operation) 을 수행한다.

> Insert / Delete / Update → Master DB

> Read → Slave DB

사용자가 많아질수록 읽기 연산의 비중이 쓰기 연산 보다 높다. 따라서 Master DB 수가 Slave DB 수보다 높은게 일반적이다.

Master-Slave 다중화는 자체는 다음과 같은 이득이 있다.

- Query 가 분산될 수 있다. (Read, Write)
- 만약 Slave 가 망가지면 다른 Slave 로 대체할 수 있고, 모든 Slave 가 망가지면 Master 가 Slave 가 복구될 때 까지 Read 를 맡으면 된다. → 가용성(Availability)

## 5. 캐시 (Cache)

캐시는 꽤 비용이 드는 연산 또는 자주 참조되는 데이터를 메모리에 두고, 보다 빨리 처리될 수 있도록 하는 저장소를 뜻한다.

서비스의 성능은 DB 를 얼마나 자주 접근하느냐에 직접적으로 관여한다. 캐시는 DB 접근을 최적화한다는 점에서 서비스 성능을 끌어올린다.

DB Layer 앞단에 Cache Layer 를 두는 방식을 살펴보자.

![4.png](/images/TSK-20/4.png)

- 만약 요청받은 데이터가 Cache 에 적재되어있는지 확인한다. 만약 적재되어있다면, DB 를 거치지않고 Web Server 로 응답이 가능하다.
- Cache 에 적재된 데이터가 없다면, DB 에서 데이터를 응답받고, Cache 저장소에 저장한다. 그 다음 Web Server 에 응답한다.

DB 데이터를 Cache 에 저장한 후 웹 서버에 전달하는 것을 읽기 주도형 캐시 전략(Read-Through Caching Strategy) 라고도 하는데. 다양한 캐시 전략이 존재한다. 캐시에 저장할 데이터의 종류, 크기, 접근 패턴을 고려해서 캐시 전략을 선정할 수 있다.

캐시 서버는 일반적으로 다양한 프로그래밍 언어로 API 를 제공하므로, 비교적 연동이 단순하다.

캐시를 사용할 때, 아래 사항들을 고려해보면 좋다.

- 갱신은 별로 없지만, 참조가 빈번한가?
- 캐시는 영속적인 데이터를 저장하지않는다. (캐시 서버가 재시작되면 모든 데이터는 사라진다.)
- 캐시 만료 정책을 마련한다. 만료 정책이 없으면 캐시가 계속 남고, 짧으면 DB 를 자주 접근하는 상황과 비슷해진다. 또 너무 길어도 DB 에 있는 원본 데이터와 차이가 날 가능성이 존재한다.
- DB 데이터를 갱신하고 + 캐시 데이터를 갱신하는 연산이 단일 트랜잭션으로 처리되어야한다. → 데이터 일관성(Data Consistency)
- 캐시 서버를 한 대만 두면, 단일 장애 지점(Single Point of Failure, SPOF) 가 발생한다. 서버 분산을 고려해볼 수 있다.
- 메모리는 얼마나 잡을 것인지? 캐시 메모리가 작으면 데이터가 캐시에서 자주 밀려난다. (Data Eviction) 자주 밀려나면 DB 접근이 빈번해지고 성능이 떨어진다. 캐시 메모리를 과할당(Overprovision) 하면 갑자기 늘어날 데이터에 대응할 수 있다.
- 캐시에 데이터가 꽉 찼을 때, 어떤 데이터를 Eviction 할 건지 정책을 마련하면 좋다.
  - LRU(Least Recently Used), 최근에 쓰이지않은 데이터 → 쓰인지 오래된 데이터
  - LFU(Least Frequently Used), 사용 빈도가 낮은 데이터
  - FIFO(First Input First Out), 캐시로 들어온 순서대로

## 6. CDN (Contents Delivery Network)

CDN 은 정적 컨텐츠를 전송하는 데에 쓰이는, 지리적으로 분산된 서버 네트워크다. 이미지, 비디오, CSS, JavaScript 등을 캐시할 수 있다.

SPA 아키텍처로 많이 쓰이는, 동적 컨텐츠 캐싱은 요청 경로나 쿼리스트링, 요청 헤더 등의 정보에 기반해서 HTML 페이지를 캐시하는 방식을 말한다.

CDN 의 동작은 다음과 같다.

1. 사용자가 사이트에 방문하면, 가장 가까운 CDN 가 정적 컨텐츠를 전달한다.
2. 만약 캐시에 없는 경우 Origin 서버에서 컨텐츠를 가져온다.
3. Origin 은 CDN 에 컨텐츠를 전달하고, 응답 헤더에 얼마나 오래 캐시 될 수 있는 지 TTL (Time to Live) 값을 설정한다.
4. CDN 은 사용자에게 컨텐츠를 반환하고, TTL 까지 캐시 Hit 된다.
5. 다른 사용자가 동일한 이미지를 요청한다면, TTL 시간만큼 CDN 에서 컨텐츠를 대신 반환하게 된다.

CDN 사용 시 고려해야할 사항이 있다.

- CDN 도 DNS 처럼 Third Party 에 의해 운영되며 데이터 전송양만큼 비용을 지불한다. 만약 컨텐츠가 자주 쓰이지 않으면 CDN 사용을 고려해봐야한다.
- TTL 값을 적절하게 설정한다. 캐싱 시간이 길면, 원본과 다를 수 있고 너무 짧으면 원본에 자주 접근할 수 있다.
- 컨텐츠 무효화 (Invalidation) 방법을 사용해서 CDN 컨텐츠를 제거할 수 있다.
  - CDN 서비스에서 제공하는 API 로 컨텐츠를 무효화
  - 컨텐츠마다 버저닝하여 업데이트 된 컨텐츠를 서빙하는 방식

이후 **[시스템 디자인] 사용자 규모와 확장성 2** 에서는, (1) 트래픽 레이어 (2) 데이터 레이어를 수평적으로 확장시키는 방법에 대해서 알아볼 것이다. 그리고 정량적인 수치를 통해 규모를 추정하는 방식도 알아본다.
